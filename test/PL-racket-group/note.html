PL-racket-group 一书的思路

这是一本采用“自底向上”构建的解释器流派PL入门书籍，与EoPL的写作思路稍有不同。作者强调一种通过错误和不足来学习的重要性，拒绝“一步到位”的教学方法。

重点关注我们的解释器是如何从“原始”演变成“高级”的，每一步“进化”是由于遇到了什么样的问题，还是说根据直觉来看需要加入（或去掉）什么新的东西等等...切勿纠结名词和定义，要知其所以然。

作者开门见山地点明了parser在整个解释器实现过程中的次要地位，并用最快捷简便的方式实现了一个从s-expression到自定义AST（ArithC，仅arithmetic）的parser。
通过parser把s-expression转化成带有类型标记的abstract syntax tree（AST），根据这个AST可以做pretty-print、compilation、verification、evaluation等工作了
书中高频出现的reduce这个词，我倾向于把它翻译为“归约”。
从最基本的arithmetic开始介绍解释器的概念，并写了第一个解释器（仅有“+”、“*”两种操作）。
通过对“+”操作符的“meaning”的疑问，引入semantics（语义）的概念：semantics is the mapping of syntax to meaning，语义是语法到“含义”的映射。
书中提出了core syntax/language和surface syntax/language的概念。
简单举了binary/unary minus的例子介绍了“语法脱糖”（desugar）的概念。
建议引入conditionals和boolean type（还未实现）。
引入function的概念，先限定了“定义函数”的位置只能在类似Dr.Racket的“definitions window”里，而“使用函数”的位置只能在“interaction window”（类似REPL）里，作者承诺接下来的章节会逐渐“解除封印”。据此限制，“运行”程序首先将“载入函数定义”（a set of definitions），我们的解释器函数也因此加入了一个新的参数（list of function definitions）。
注意：list是有序结构，而在此我们并不需要这堆函数定义的排列顺序，出于其他因素考虑，可以用其他数据结构来代替。
先只考虑单参函数，解构函数的三个部分（函数的名字、formal parameter、body）。为了考察body，引入了identifier（后面会延拓成variable）和function application的概念。
注意“函数定义”和“函数调用”的区别，前者用于描述一个函数，而后者则是使用一个函数。
函数调用时actual parameter和formal parameter的最“单纯直白”的交互关系——search-and-replace（substitution）。

  • 在用substitution实现函数调用的解释器里，会有一个问题：参数如果是复杂的表达式，那么是要先evaluate再substitute还是先substitute再evaluate？前者叫做eagar evaluation（deferred substitution），后者叫做lazy evaluation（deferred evaluation，或者叫“eagar substitution”?），这两者是semantics层面的区别

  • 在用substitution实现的解释器里还有另一个问题：在正式interpret之前要traverse源程序几遍？能否只traverse那些将要被evaluated的部分？试想：substitution会traverse所有分支语句的branches，而解释的时候并不会这样

  • substitution导致了解释器对源程序的依赖（会rewrite the program source）。而对于编译器，没有必要如此依赖源码（parser出来AST之后基本就独立了），所以从这个角度看，编译器是一个更加“portable”的东西

出于以上几点考虑，substitution其实并不是一个实现函数调用的好选择。

因此我们引入“环境”（environment）概念，作为substitution的“升级版”。在环境这个结构中储存的name-value bindings可以理解为是一种“intent to substitute”，即上面提到的“deferred substitution”（先evaluate好然后存起来，等真正需要的时候再“substitute”）。

注意：从substitution到environment的升级，我们只改变了语言的“实现策略”（implementation strategy），而语言本身并未改变，即表示程序的数据类型和解释结果不变！

在substitution-based解释器中，当且仅当函数调用时会产生新的substitution；同理在environment-based解释器中，当且仅当函数调用时会改变environment。（如何改变？）
解释器升级到environment-based之后会引入新的问题——dynamic scoping。它会使环境随着程序的执行（函数的调用）不断“增长”，而从不“衰减”，其结果是，某个identifier是否被绑定取决于程序执行（函数调用）的历史，这毫无疑问是一个程序语言设计的糟粕（如lisp machine lisp 和emacs lisp）。substitution-based解释器和实现了lexical scoping（static scoping）的environment-based解释器则避免了这个问题。
(define (f1 x) (f2 4))
(define (f2 y) (+ x y))
; what will happen when interpreting (f1 3)?
lexical（static）的含义是，“可以根据源代码确定”、“毋需运行程序就知道”，而dynamic就是相反的意思。
值得注意的是，到此为止，的解释器还未实现first-class function，函数的定义和调用还是严格分离的，所以还未提及Closure的概念，但即使如此也已经有区别dynamic和lexical的方法了，尽管代码上的区别非常微妙！
Top-Level Scope的情况有所不同：
(define y 1)
(define (f x) (+ x y))
(define y 2)
; (f 10)
; -> 12
(define y 1)
(define f (let ([z y]) (lambda (x) (+ x y z)))
(define y 2)
合理的做法是，尽量避免这种重定义。

Programming languages should be designed not by piling feature on top of feature, but by removing the weaknesses and restrictions that make additional features appear necessary.
程序语言的设计并非堆砌特性，而是去除可能会导致人们引入“新特性”的缺点和限制。

尝试思考并回答以下两个问题：

  • 函数定义能不能是一个表达式呢？
  • 如果函数定义是表达式，那么它的“值”是什么东西？
一个简单的回答是，函数定义的值是一个“function value”，它显然不能像number一样，可以用“+”、“*”来操作，但是我们却可以把这种“function value” apply给arguments，所以我们可以尝试让“function value”出现在函数调用的“函数名”位置，比如：
(+ 2 ([define (f x) (* x 3)] 4))
; -> (+ 2 (* 4 3)) -> 14
接下来我们重点考察函数定义在“调用时”的情形，即所谓的immediate functions。加入了function value之后，“值”的种类扩展了（之前一直只有number一种），相应的，解释器产生的结果种类也会被扩展。
之前直接借用宿主语言（host language）的“+”和“*”也需要重新定义，以适应不止一种“值”类型的情况。
问题来了，由于任何表达式都可能出现在函数调用的“函数名”位置，如何确定这个表达式“真的”是一个函数定义呢？是语法符合？还是被解释器解释为function value（作者选了这种）？还是其他标准？
嵌套函数定义的情况会出问题，function value不能仅仅只是name、argument和body，还应记住已经应用的substitution，因此引入了Closure概念，同时抛弃name——函数天生是匿名的，我们要对“函数定义”和“函数命名”加以区分。
实现了Closure之后，原来用于实现lexical scoping的方法可以改一改，现在可以在函数调用的时候直接扩展caller的环境而不是扩展一个新建的环境了。（为什么？）
实现了Closure的environment-based解释器自动实现了capture-free substitution：
; “幼稚”的substitution
(lambda (f)
(lambda (x)
(f 10)))
; f -> (lambda (y) (+ x y)) 这里x是free的
(lambda (x)
((lambda (y) (+ x y)) 10))
; 经过“幼稚”的substitution后，x居然被绑定（抓住）了！
; capture-free substitution
(lambda (f1)
(lambda (x1)
(f1 10)))
; f -> (lambda (y1) (+ x y1))
(lambda (x1)
((lambda (y1) (+ x y1)) 10)) ; x还是free的
; 把所有原先bound的identifier重命名，free的保持不变
; ，再进行substitution，这样就不会被“意外抓住”了

实现了Closure的env-based解释器呢，由于function value（或者叫做closure value）包含了env，这个env里面只储存了当前函数定义处“看得见”的绑定，而“内层”的“同名identifier”是不会影响的，也就是说与函数调用的结果与它在何处被调用无关，也就是lexical scoping的含义。

the top-level has an “infinite scope”，考察下面三个例子：
(define (double x) (+ x x))
(define (quadruple x) (double (double x)))
(quadruple 10)
(let ([double (lambda (x) (+ x x))])
(let ([quadruple (lambda (x) (double (double x)))])
(quadruple 10)))
(let ([quadruple (lambda (x) (double (double x)))])
(let ([double (lambda (x) (+ x x))])
(quadruple 10)))
caller和callee分别代表了什么？我的理解是caller是某一片程序的context，callee就是被调用的函数（有待商榷）
到此为止，给解释器特定输入，总会得到相同的值。
我们想要表达某个“改变”“之前”和“之后”的区别。
因此我们引入“时间”的概念。
(define new-loc
(let ([n (box 0)])
(lambda ()
(begin
(set-box! n (add1 (unbox n)))
(unbox n)))))
(define new-loc-broken
(lambda ()
(let ([n (box 0)])
(begin
(set-box! n (add1 (unbox n)))
(unbox n)))))
比较以上两个函数的区别，把第一个正确实现的counter函数中的let“脱糖”，可以得到：
(define new-loc
((lambda (n)
(lambda ()
(begin
(set-box! n (add1 (unbox n)))
(unbox n))))
(box 0)))

只要调用new-loc一次(box 0)就被绑定到n了，接下来继续调用new-loc并不会重新创建box。

从这个例子我们可以看到，我们希望new-loc中的n始终是“同一个”box，而同时又希望这个box中的值一直在变化：it must be lexically the same, but dynamically different!

问题是，怎样保证解释器先后对同样的表达式求出了不一样的值呢？最直觉的猜测就是，解释器函数另外一个参数——环境，有所变化。

另外，为了达到这一目的（使得解释器对同一表达式先后产出不同的值），我们还需要记录“上一次”解释时留下的“变化”，再把这些“变化”告诉“下一次”解释：

; interp : ExprC * Env -> Value * Env

然而值得注意的是：环境已经被设计用于记录deferred substitution了。某种意义上讲，环境是repository of lexical scope information。假如我们让环境从一个表达式里跑进去又跑出来，再跑进另一个表达式里……就会乱套。

所以不能用环境来表达解释器对同一表达式“先后”求值的区别。

数学里的“变量”和计算机里的“变量”有什么异同？

  • 数学语言里的“变量”指的是在对函数不同的“唤醒”（invocations）中能不断变化的东西，所以也叫“自变量”
  • 编程语言里的“变量”（variable）是在同一个scope下（或者说，对函数的同一次“唤醒”中）可以不断改变值的东西

主流编程语言overload了mutation的syntax（用=或者:=等），在Scheme里要求写代码时显式地加以区分“绑定”和“赋值”（用set!）：

  • 声明——建立新的绑定，并加入环境（bind [name : symbol] [location : number]）
  • 赋值——找到变量名所绑定的location，并改变这个location存储的value
  • 声明+赋值——建立新的name-location绑定，并“几乎同时”完成对应的location-value存储
  • 事实上“绑定”和“存储”的行为模式很相似，但为了对应不同的过程和功能需要加以区分

关于name，location，value三者之间的关系，可以参考Python里的变量名、变量的id，变量的值。顺便一提Python3里的id()函数，我认为基本与本书提到的lookup函数功能相同，都是根据一个“名字“在环境里查找对应的location（address）。

关于参数传递的问题

如果每次函数调用都给函数的参数（“形参”）分配一个新的location，然后在这个location中存入“实参”reduce出的值，那么“形参”所在的location就会和“实参”所在的location不同（事实上，我们的解释器根本没有关注“实参”的location），考察以下程序：

(let ([f (lambda (x) (set! x 3))])
(let ([y 5])
(begin
(f y)
y)))

如果按照上述处理参数的做法，结果会是5而非3！现在设想，假如“实参”也是一个变量，有自己的location和value，我们是不是可以直接把location传给“形参”（“形参”和“实参”形成了variable aliases），而不是给这个value重新分配一个location（然后绑给“形参”）呢？

这样的话，在函数体内“形参”值的变化，就会以相同的location为“媒介”，“流回”到调用这个函数的上下文，也就是“实参”的值跟着变化了——这种传参策略被叫做“call-by-reference”，或者根据本书作者的命名，我也可以叫它“call-by-location”;)

这种传参策略事实上并不被现在主流语言广泛采用，由于它有一下的缺陷：

  • 程序员可能会在caller中不小心alias了一个变量并改变了它的值（被alias的另外的变量也随之改变了，这显然不是我们想要的），这会引起一些非常微妙的，难以察觉到错误；
  • “call-by-value”其实不一定需要copy large data structure，它能兼容仅传递large data structure的地址（如何做到的？）
  • ...

如果要“call-by-reference”，那么应当让caller来决定是否传递reference，即是否愿意让callee“分享”caller变量的memory address（就像C/C++那样），但尽管如此，callee也会被“蒙在鼓里”，它并不知道自己的变量是否被alias了。

如果caller想要“邀请”callee来给自己的变量做一些改变，可以考虑给callee传递一个boxed value，让callee弄好再返回出来。

三种传参策略粗略理：

  • call-by-name，可以理解为最开始的substitution-based解释器实现的策略
  • call-by-reference，使用variable alias让实参和形参的location相同
  • call-by-value，调用函数时给形参分配新的location，并存入实参的值

递归和Cycles

递归是一种自我指涉（self-reference）的行为。当我们谈及递归时，一般有两种含义：

  • recursion in data
    • referring to sth. fo the same kind
    • referring to the same thing itself
  • recursion in control (i.e. functions)

树是有限的递归数据结构，图（graph）常常是cyclic的数据结构。

创建递归三步走：

  • 建立一个容器（可以是box，也可以是变量）
  • 建立一个指向这个容器的reference
  • 在使用之前把容器的内容变成这个reference

如下例子所示，我们成功建立了一个递归函数fact

(let ([fact (box 'dummy)])
(let ([fact-fun
(lambda (n)
(if (zero? n)
1
(* n ((unbox fact) (- n 1)))))])
(begin
(set-box! fact fact-fun)
((unbox fact) 10))))

事实上，可以不需要fact-fun的帮助：

(let ([fact (box 'dummy)])
(begin
(set-box! fact
(lambda (n)
(if (zero? n)
1
(* n ((unbox fact) (- n 1))))))
((unbox fact) 10)))
还可以用变量来做容器：
(let ([fact 'dummy])
(begin
(set! fact
(lambda (n)
(if (zero? n)
1
(* n (fact (- n 1))))))
(fact 10)))

Type checking

静态类型检查（static type checking）就是在正式运行程序之前“空转”一遍代码，检查类型是否符合等等：

(define (f [n : number]) : number

  (begin

    (display n)

    (+ n 3)))

(define (f [n : number]) : number

  (begin

    (display n)

    (+ n 3)))

对两段代码同时执行(f "x")，前者不会执行display，后者会，说明后者这种没有type annotation的程序仅在“运行时”（runtime）才会察觉出类型不匹配这种错误。

但是即使是没有类型标注的语言，如果有语法错误，也会在正式运行之前被parser察觉到。事实上，parser可以被看作是一个“弱化版”type checking——它只检查“上下文无关”（context-free）的语法，而type checking检查的是“上下文敏感”（context-sensitive）语法。两者共同点就是，它们都用某种方式“迫使”程序符合某些规则，否则就“休想运行”;)

+，*以及任何consume两个number的函数，type checking的结果基本是一样的，与eval不同，tc对表达式的“值”是什么“不感兴趣”，而只对表达式的type感兴趣。从这个角度考虑，在正式解释运行之前通过tc“空转”一遍代码的时间要远小于正式解释运行的时间。

总之写type checking的要领就是，声称的类型要和实际的类型一样，如函数的返回类型声称，要和代入参数类型后实际检查body出来的类型一致才行
